# Performance / Gantt Guide

The Performance screen shows the time distribution and processing steps of the
last run in a Gantt-style view. The goal is to quickly identify a slow model,
OCR or file-processing step.

## Authorization

| Route | Guard |
| --- | --- |
| `/performance` | `canRun` |

## Data source

| Endpoint | Purpose |
| --- | --- |
| `GET /gantt/latest` | Fetches Gantt data for the most recent session. |
| `GET /gantt/{session_id}` | Fetches Gantt data for a specific session. |

On the backend, timing data is written during processing and includes
per-model and per-worker durations.

## User flow

1. Complete a run on the Run screen.
2. Open the Performance screen.
3. Verify that the latest Gantt data has loaded.
4. Use the `Models` or other detail filters to find the slow step.
5. If the issue is model-related check the Settings/model file; if OCR-related check the backend logs.

## Interpretation

| Symptom | Meaning |
| --- | --- |
| A single model step takes too long | Check GPU/CPU, the model file or TensorRT fallback. |
| OCR steps are long | Inspect image quality, number of rows or OCR worker performance. |
| First batch long, next batches normal | May be model warm-up or cache effect. |
| All batches slow | Could be hardware, batch size or disk I/O problem. |

## Operation notes

- The Gantt screen is a debugging tool; it is for performance analysis, not a user report.
- Before a release, run a timing comparison with the same test dataset.
- Settings `batch_size`, model type and `PREFER_TRT` behavior can affect performance.

## Expected screenshots
