The Performance screen shows the time distribution and processing steps of the
last run in a Gantt-style view. The goal is to quickly identify a slow model, OCR
or file-processing step.

<Info>
  Users with the `canRun` permission can access this screen (`/performance`).
</Info>

## Data source

| Endpoint | Purpose |
| --- | --- |
| `GET /gantt/latest` | Fetches Gantt data for the most recent session. |
| `GET /gantt/{session_id}` | Fetches Gantt data for a specific session. |

On the backend, timing data is written during processing and includes per-model
and per-worker durations.

## User flow

<Steps>
  <Step title="Complete a run">
    Complete a process on the Run screen.
  </Step>
  <Step title="Open Performance">
    Open the Performance screen and verify the latest Gantt data has loaded.
  </Step>
  <Step title="Find the slow step">
    Use the `Models` or other detail filters to find the slow step.
  </Step>
  <Step title="Drill into the source">
    If the issue is model-related check the Settings/model file; if OCR-related check the backend logs.
  </Step>
</Steps>

## Interpretation

<AccordionGroup>
  <Accordion title="A single model step takes too long" icon="gauge">
    Check GPU/CPU, the model file or TensorRT fallback.
  </Accordion>
  <Accordion title="OCR steps are long" icon="gauge">
    Inspect image quality, number of rows or OCR worker performance.
  </Accordion>
  <Accordion title="First batch long, next batches normal" icon="gauge">
    May be model warm-up or cache effect.
  </Accordion>
  <Accordion title="All batches slow" icon="gauge">
    Could be hardware, batch size or disk I/O problem.
  </Accordion>
</AccordionGroup>

## Operation notes

<Note>
  - The Gantt screen is a debugging tool; it is for performance analysis, not a user report.
  - Before a release, run a timing comparison with the same test dataset.
  - Settings `batch_size`, model type and `PREFER_TRT` behavior can affect performance.
</Note>
