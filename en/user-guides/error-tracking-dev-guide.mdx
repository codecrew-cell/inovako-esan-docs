The Error Tracking screen provides operational error tracking for users with the
developer role. Backend exceptions and errors reported by the frontend are kept in
`errors.db`.

<Info>
  Users with the `developer` role can access this screen (`/error-tracking`). On
  the backend, the developer check depends on the `_check_developer` behavior in
  `Api/dev_error_mixin.py`.
</Info>

## Data source

| Source | File / endpoint |
| --- | --- |
| Error DB | `errors.db` |
| Backend capture | `log/error_tracker.py` |
| Frontend capture | `POST /errors/frontend` |
| Developer API | `/dev/errors`, `/dev/issues`, `/dev/session-stats` |

## Screen flow

```mermaid
flowchart TD
    A[Error occurs] --> B{Source}
    B -->|Backend| C[error_tracker.capture]
    B -->|Frontend| D[POST /errors/frontend]
    C --> E[errors.db]
    D --> E
    E --> F[Filter on Error Tracking screen]
    F --> G[Issue/fingerprint analysis]
```

## Endpoints

| Endpoint | Purpose |
| --- | --- |
| `GET /dev/errors` | Lists errors with source, level, fingerprint, since and search filters. |
| `GET /dev/errors/{error_id}` | Returns the detail of a single error. |
| `DELETE /dev/errors` | Clears the error list. |
| `GET /dev/issues` | Returns the issue list grouped by fingerprint. |
| `PATCH /dev/errors/{error_id}/status` | Updates the error status. |
| `GET /dev/session-stats` | Returns session/error statistics for the last 30 days. |

## Review standard

<Steps>
  <Step title="Check the issue view">
    First check repeating fingerprints in the `Issues` view.
  </Step>
  <Step title="Inspect a single error">
    In the error detail, inspect the endpoint, stack trace, username and session id fields.
  </Step>
  <Step title="Drill into the source">
    For a backend error find the related FastAPI endpoint; for a frontend error check the route and component info.
  </Step>
  <Step title="Update the status">
    If resolved, update the status or clear the list after a release.
  </Step>
</Steps>

## Retention policy

<Note>
  `log/error_tracker.py` can automatically clean up old errors. By default, errors
  older than 30 days are deleted. Critical errors should be moved to the release
  notes or an issue tracker.
</Note>
