The Dashboard is designed so that, after logging in, the user can see the platform
status and recent analysis summaries on a single screen. The login screen
validates the username, starts a new browser session and routes the user to the
Run screen.

## Login flow

<Steps>
  <Step title="Enter a username">
    Enter your username and log in.

    <Frame caption="Login screen">
      ![Login](/assets/screens/login.png)
    </Frame>
  </Step>
  <Step title="Authentication">
    The frontend calls `GET /ldap_auth/{username}` to validate the user.
  </Step>
  <Step title="Session creation">
    On success, the browser creates a `login_session_id` and the session is
    recorded on the backend via `POST /sessions/start`.
  </Step>
  <Step title="Navigate to Run">
    The application clears stale analysis state and navigates to the Run screen.
  </Step>
</Steps>

<Note>
  During login, the application stores `login_username` and `login_session_id` in
  `sessionStorage`. These values are correlated with performance logs, the Gantt
  view, error tracking and session recovery flows.
</Note>

## Dashboard metrics

`GET /dashboard/stats` powers the dashboard cards. If the backend does not
respond, the screen opens with safe default values.

| Field | Meaning |
| --- | --- |
| `total_core_length` | Total processed core length. |
| `recovery_rate` | Recovery percentage. |
| `class_counts` | Class or lithology distribution. |
| `fault_zones` | Detected fault or special zones. |
| `active_models` | Number of active models. |
| `ai_confidence` | Summary confidence metric. |
| `system_status` | Backend/service status summary. |
| `recent_scans` | Recent analysis records. |
| `timeline_items` | Recent operational events. |
| `map_sites` | Location cards and site map data. |

## Screen panels

The Dashboard is not a job-launch screen; it is for monitoring and summarizing the
overall operation status. It answers "what is happening in the system right now?".
All data is loaded in real time via `GET /dashboard/stats`.

<CardGroup cols={2}>
  <Card title="Hero area" icon="layout-panel-top">
    System Status (Online/Offline), a personalized greeting, the active site, the total core length processed during the day, and the Shift Date.
  </Card>
  <Card title="KPI cards" icon="gauge">
    Total Analyzed, Avg Core Recovery, Detected Features and Active Models metrics.
  </Card>
  <Card title="Drilling Progress Map" icon="map">
    Summary view of site and drillhole points (active/completed/planned); "View Full Map" for detail.
  </Card>
  <Card title="Recent Core Scans" icon="table">
    Recently uploaded/processed core records: File Name, Depth, Status and a VIEW action.
  </Card>
  <Card title="AI Model Health" icon="activity">
    AI Confidence (%) and System Load (%) — shown with progress bars.
  </Card>
  <Card title="Live Feed" icon="rss">
    Recent events in chronological order: time, title, description and event type (normal/warning).
  </Card>
</CardGroup>

<Note>
  The Dashboard is monitoring/assessment-focused, not for launching jobs. For the
  definitive project record, rely on the outputs and export files from the relevant
  analysis screen.
</Note>

## Location behavior

<Tip>
  If the browser permits geolocation, the Dashboard attempts reverse geocoding
  through OpenStreetMap Nominatim. If not available, it falls back to `N/A` or
  default location data.
</Tip>

## Operation notes

- Repeated clicks on the login button are prevented; a new request is not started while one is active.
- During login, the previous Redux analysis state and UI cache records are cleared.
- If `BASE_URL` is not defined, the frontend defaults to `http://localhost:8002`.
- Dashboard data is for decision support; for the definitive project record, rely on the outputs and export files from the relevant analysis screen.
